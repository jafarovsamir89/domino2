Все задачи рекомендуется делать в таком порядке:

1. Безопасность и транспорт.

### 1.2. Обязательные правила перед каждым исправлением

Перед изменением любого пункта агент должен:

1. Прочитать все связанные файлы и найти все места использования проблемного кода.
2. Проверить, дублируется ли логика в `js/` и `www/js/`.
3. Убедиться, что исправление не ломает multiplayer, solo, team mode и platform flow.
4. Если затрагиваются токены, ставки, очки, room restore или auth, исправление должно сопровождаться ручной проверкой сценариев.
5. После правки выполнить минимально необходимую проверку:
   - `npm test`
   - `npm run build -w @domino2/api`
   - `npm run build -w @domino2/admin`
   - при изменении legacy server: `node --check server/index.js` и `node --check server/DominoRoom.js`

### 1.3. Важная особенность проекта

У проекта смешанная архитектура:

- legacy game client в `js/`
- копия web-ассетов в `www/`
- legacy game server в `server/`
- новая platform API в `apps/api`
- admin panel в `apps/admin`
- shared/db packages в `packages/*`

Из-за этого одна и та же проблема может существовать в двух контурах. Особенно это касается:

- auth secrets
- realtime presence
- клиентских ассетов
- сетевых URL
- аутентификации игровых токенов

## 3. Блок безопасности

## 3.1. Убрать insecure fallback для `BETTER_AUTH_SECRET`

### Приоритет
P0

### Затронутые файлы

- [server/platformAuth.js](/abs/path/c:/domino2/server/platformAuth.js:1)
- [apps/api/src/modules/auth/game-token.ts](/abs/path/c:/domino2/apps/api/src/modules/auth/game-token.ts:1)
- [apps/api/src/modules/auth/better-auth.config.ts](/abs/path/c:/domino2/apps/api/src/modules/auth/better-auth.config.ts:18)

### Проблема

Сейчас проект использует небезопасный fallback:

- `process.env.BETTER_AUTH_SECRET || "change-me"`

Это означает, что если переменная окружения забыта, сервер не падает и продолжает работать на заранее известном секрете. Любой, кто знает исходный код, может:

- подписывать валидные токены
- подделывать игровые сессии
- создавать фальшивые `game token`
- обходить авторизацию
- потенциально вмешиваться в экономику матчей

### Почему это особенно опасно здесь

Проблема существует не в одном месте, а минимум в трех:

1. legacy-проверка игровых токенов
2. platform API
3. better-auth конфигурация

Если исправить только один файл, система все равно останется дырявой или станет несовместимой между клиентом, legacy server и platform API.

### Как исправлять правильно

Нужно создать единый подход:

1. Вынести чтение секрета в общую безопасную функцию.
2. Убрать fallback на `"change-me"` и другие тестовые строки.
3. При отсутствии секрета сервер должен завершать запуск с понятной ошибкой.
4. Документировать обязательность переменной в `.env.example`, deploy scripts и README.

### Безопасный способ внедрения

1. Создать маленькую helper-функцию, например `readRequiredSecret(name)`.
2. Проверять:
   - переменная существует
   - строка не пустая
   - длина не меньше 32 символов
   - значение не равно `"change-me"`, `"replace-me"`, `"secret"`, `"test"`
3. При нарушении выбрасывать `Error` во время boot.
4. Использовать одну и ту же функцию в:
   - `server/platformAuth.js`
   - `apps/api/src/modules/auth/game-token.ts`
   - `apps/api/src/modules/auth/better-auth.config.ts`

### Чего нельзя делать

- Нельзя просто заменить `"change-me"` на `"replace-me"`.
- Нельзя генерировать секрет динамически при старте сервера. Это сломает уже выданные токены.
- Нельзя держать разные секреты для валидации одного и того же типа токенов без явного versioning.

### Что еще улучшить

- Добавить лог при старте: secret loaded, но без вывода самого значения.
- Добавить health/startup check, который валидирует обязательные env-переменные до поднятия HTTP-сервера.

### Как проверить после исправления

1. Запустить API без `BETTER_AUTH_SECRET` и убедиться, что сервис завершается с понятной ошибкой.
2. Запустить API с валидным секретом и проверить логин.
3. Сгенерировать `game token` и убедиться, что legacy server его принимает.
4. Поменять секрет и убедиться, что старые токены перестают быть валидными.

## 3.2. Перевести трафик на HTTPS и запретить cleartext

### Приоритет
P0

### Затронутые файлы

- [scripts/gcloud/nginx-domino2.conf](/abs/path/c:/domino2/scripts/gcloud/nginx-domino2.conf:1)
- [js/network.js](/abs/path/c:/domino2/js/network.js:42)
- [capacitor.config.json](/abs/path/c:/domino2/capacitor.config.json:1)
- [android/app/src/main/AndroidManifest.xml](/abs/path/c:/domino2/android/app/src/main/AndroidManifest.xml:4)
- [android/app/src/main/res/xml/network_security_config.xml](/abs/path/c:/domino2/android/app/src/main/res/xml/network_security_config.xml:1)

### Проблема

Сейчас:

- Nginx слушает только `80`
- клиент имеет HTTP fallback
- Android явно разрешает cleartext
- Capacitor использует `androidScheme: "http"`
- смешанный контент разрешен

Это открывает MITM-риски, утечки токенов, утечки игровых событий и проблемы с cookie/security policy.

### Как исправлять правильно

Нужно закрывать проблему на всех уровнях сразу:

1. Внешний reverse proxy.
2. Web client URL defaults.
3. Platform origins.
4. Android manifest и network security.
5. Capacitor config.

### Безопасный план

#### Шаг 1. Настроить TLS в Nginx

Нужно:

- добавить `listen 443 ssl http2`
- подключить `fullchain.pem` и `privkey.pem`
- сделать редирект `80 -> 443`

#### Шаг 2. Убрать HTTP как дефолт в клиенте

В `js/network.js` нужно:

- отказаться от `http://34.28.23.216:2567` как постоянного fallback
- перевести серверные URL на доменное имя
- делать `https://...` как дефолт для web
- использовать `wss://` или `https://`-совместимый endpoint там, где это поддерживается Colyseus

#### Шаг 3. Исправить Android и Capacitor

Нужно:

- в `capacitor.config.json` перейти на `https`
- убрать `allowMixedContent: true`
- убрать `android:usesCleartextTraffic="true"`
- поменять `network_security_config.xml` так, чтобы cleartext был запрещен

### Риски при внедрении

- Можно случайно сломать локальную разработку.
- Можно сломать мобильную сборку, если backend локально недоступен по HTTPS.

### Как не повредить систему

Нужно разделить окружения:

- `development` может использовать localhost исключения
- `production` должен строго требовать HTTPS

Лучше завести env-флаги:

- `PUBLIC_GAME_URL`
- `PUBLIC_PLATFORM_API_URL`
- `PUBLIC_ADMIN_URL`

И использовать их вместо хардкода IP.

### Проверка

1. Web открывается по `https://`.
2. Все API и realtime-запросы идут без mixed content warning.
3. Android release build подключается без cleartext исключений.
4. Токены и логин работают после перехода на HTTPS.

## 3.3. Закрыть XSS во всех пользовательских именах и динамических вставках

### Приоритет
P0

### Затронутые файлы

- [js/renderer.js](/abs/path/c:/domino2/js/renderer.js:435)
- [js/app.js](/abs/path/c:/domino2/js/app.js:1629)
- [js/app.js](/abs/path/c:/domino2/js/app.js:1693)
- [js/app.js](/abs/path/c:/domino2/js/app.js:1763)
- [js/app.js](/abs/path/c:/domino2/js/app.js:1814)

### Проблема

Пользовательские имена, friend names, invite names и другие данные из сети вставляются через `innerHTML`.

Это позволяет внедрить HTML/JS-пейлоад, например:

```html
<img src=x onerror=alert(1)>
```

или более опасные варианты для кражи токена из `localStorage`.

### Почему это критично

Потому что в проекте platform game token хранится в `localStorage`. Значит, любая XSS уязвимость может быть превращена в:

- кражу токена
- перехват игровой сессии
- отправку действий от имени пользователя
- финансовые злоупотребления

### Как исправлять правильно

Нужно менять не строку в одном месте, а сам подход к построению UI:

1. Для пользовательского текста использовать `textContent`.
2. Для сложных карточек строить DOM руками через `createElement`.
3. Допускать `innerHTML` только для полностью статического шаблона без пользовательских полей.

### Рекомендуемая стратегия

#### Подход A. Полный отказ от `innerHTML` для динамического UI

Для каждого проблемного блока:

- создать контейнер
- создать `strong`, `span`, `div`
- заполнить `textContent`
- собрать дерево через `appendChild`

#### Подход B. Если нужен HTML-шаблон

Можно добавить `escapeHtml(text)`, но только как временную меру. Правильнее все равно использовать DOM API.

### Места, которые нужно проверить дополнительно

Нужно пройти по всему фронтенду поиском:

- `innerHTML =`
- template literals с `${name}`
- template literals с `${displayName}`
- вставки room code, room mode, friend id, invite data

### Что нельзя делать

- Нельзя полагаться только на `sanitizeName()` на клиенте.
- Нельзя считать, что server-side sanitize полностью спасает.
- Нельзя экранировать только `<script>`; XSS бывает не только через script-теги.

### Дополнительное улучшение

Добавить отдельный helper-модуль для безопасного UI:

- `setText(el, value)`
- `appendLabeledValue(parent, label, value)`
- `buildPlayerChip({name, id})`

Это снизит шанс повторного появления XSS.

### Проверка

1. Создать игрока с именем вида `<img src=x onerror=alert(1)>`.
2. Проверить leaderboard, room list, friends, invites, round-end и game-over screens.
3. Убедиться, что строка отображается как текст, а не как HTML.

## 3.4. Убрать хранение чувствительных токенов в `localStorage`

### Приоритет
P0

### Затронутые файлы

- [js/account.js](/abs/path/c:/domino2/js/account.js:150)

### Проблема

`platformGameToken` читается и пишется через `window.localStorage`.

При наличии XSS это эквивалентно открытому доступу к токену.

### Что делать правильно

Нужно решить, какой токен вообще должен жить на клиенте:

1. Если это short-lived game token для join/reserve/settle, срок жизни должен быть коротким.
2. Если возможно, токен лучше держать в памяти, а не на диске.
3. Если persistent login уже обеспечивается cookie-based better-auth session, game token должен выдаваться по запросу и не жить долго.

### Безопасный вариант миграции

1. Оставить `better-auth` с `credentials: include`.
2. `platform/game-token` выдавать по действующей сессии.
3. Game token:
   - хранить в memory state
   - при reload получать заново
   - TTL сделать небольшим

### Если полная миграция пока невозможна

Временная промежуточная мера:

- минимизировать TTL токена
- очищать токен на logout, room leave, session invalidation
- не использовать его дольше необходимого

### Проверка

1. Пользователь логинится.
2. После reload клиент может заново получить game token.
3. После logout токен исчезает.
4. При истечении TTL клиент корректно запрашивает новый токен.

## 4.1. Устранить гонки `turnInProgress`

### Приоритет
P0

### Затронутые файлы

- [js/app.js](/abs/path/c:/domino2/js/app.js:2512)
- [js/app.js](/abs/path/c:/domino2/js/app.js:3123)
- [js/app.js](/abs/path/c:/domino2/js/app.js:3189)

### Проблема

Сейчас несколько веток используют `setTimeout()` и флаг `turnInProgress` без централизованного управления.

Из-за этого возможны ситуации:

1. Старый таймер срабатывает после нового `startDeal()`.
2. `playTile()` получает вызов в момент, когда флаг еще не синхронизирован.
3. AI и human flow могут пересекаться.
4. Сторонний timeout из предыдущего хода вмешивается в новое состояние матча.

### Симптомы

- тайл не играется по клику
- ход зависает
- AI не делает ход
- игра оживает только после watchdog

### Правильная цель исправления

Нужно не просто вставить `clearTimeout` в одну строку, а привести управление ходом к явной модели состояния.

### Правильный план

#### Шаг 1. Ввести явные ссылки на таймеры

В конструкторе нужны отдельные поля:

- `_firstTurnTimeout`
- `_advanceTurnTimeout`
- `_aiTurnTimeout`
- `_humanPassTimeout`

#### Шаг 2. Любой новый жизненный цикл раунда должен отменять старые таймеры

Нужно сделать helper:

- `clearRoundTimers()`

Он должен отменять все таймеры, относящиеся к текущему ходу и текущему раунду.

#### Шаг 3. Разделить понятия

Сейчас один флаг используется как защита от всего сразу. Лучше иметь:

- `turnInProgress`
- `roundTransitionInProgress`
- `aiThinking`

Или один enum state, например:

- `idle`
- `animating`
- `awaiting-human`
- `ai-thinking`
- `round-ending`

### Наиболее безопасный минимальный fix

Если глубокий рефакторинг пока слишком рискован, можно сделать промежуточный вариант:

1. В начале `startDeal()` отменять старые таймеры.
2. Перед постановкой нового timeout на первый ход отменять предыдущий `_firstTurnTimeout`.
3. В `queueAITurnIfNeeded()` хранить и отменять `_aiTurnTimeout`.
4. Перед `endDeal()` и `endRound()` очищать все таймеры.

### Чего нельзя делать

- Нельзя добавлять еще больше `setTimeout` без registry.
- Нельзя оставлять старые таймеры жить между deal/match transitions.
- Нельзя надеяться только на watchdog.

### Проверка

Нужно вручную проверить:

1. Новый матч.
2. Быстрый переход к следующей раздаче.
3. AI первый ход.
4. Human первый ход.
5. Team mode.
6. Возврат из меню.
7. Серия быстрых кликов во время анимации.

## 4.2. Исправить `playTile()` guard и связанный порядок вызовов

### Приоритет

### Проблема

## 14. Definition of Done для всего проекта

Проект можно считать заметно оздоровленным только когда выполняются все условия ниже:

1. Ни один auth-контур не использует insecure fallback secret.
2. Production трафик идет только через HTTPS.
3. В UI не осталось небезопасных `innerHTML` с пользовательскими данными.
4. Game token не хранится долгоживущим образом в `localStorage`.
5. Игровой цикл не зависит от гонок таймеров.
6. Восстановление комнаты работает на реальных model instances.
7. Open rooms/presence корректны минимум для multi-instance сценария или это явно запрещено конфигурацией.
8. Free play и coin play реально различаются по логике.
9. Есть хотя бы минимальный regression test suite.
10. Документация по deploy и env соответствует реальному состоянию кода.
