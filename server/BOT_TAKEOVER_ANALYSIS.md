# Bot Takeover — глубокий анализ по режимам и крайним случаям

> Цель документа — ничего не упустить. Читать вместе с `BOT_TAKEOVER_INTEGRATION.md`.
> Правки сильные, проект — на реальные ставки. Главное правило: ничего не сломать при выключенном флаге.

## 0. Главный инвариант (самое важное)

Захват места ботом меняет **ТОЛЬКО** поля `controller` / `takeoverActive` / `takeoverReason` / `takeoverSince`.

Он **НИКОГДА не трогает**:
- `player.userId` — иначе игрок выпадёт из `participants` и из расчёта ставок (`matchResultPayload.js` пропускает `if (!player.userId) continue`).
- `player.seatIndex` — определяет порядок ходов и команду.
- `state.playerOrder` — порядок ходов и принадлежность к команде (`i % 2`).
- `player.isBot` — остаётся `false` у захваченного человека (это всё ещё его место, бот лишь временно играет).
- `player.name`, `score`, `roundWins`, `handCount`.

Почему: это делает захват «прозрачным» для всех подсистем (порядок ходов, команды, счёт, результат матча, экономика). Меняется только «кто физически жмёт кнопки».

## 1. Различаем три типа «ботов»

| Тип | isBot | controller | userId | Участвует в participants/ставках |
|-----|-------|-----------|--------|------------------------------|
| Обычный AI-оппонент (заполнение пустых мест) | `true` | `human` | пустой | Нет |
| Человек на месте | `false` | `human` | есть | Да |
| **Захватченное место (бот за человека)** | `false` | `bot` | есть | **Да** |

Ключевое: `begin()` отказывает, если `player.isBot === true` — обычный AI нельзя «захватить». Актор хода = `player.isBot || botTakeover.isBotControlled(player)`.

## 2. Матрица режимов

Режимы (из `roomConfig.js`): `roomMode` ∈ {`ffa`, `team`}; `playerCount` 2..4; `aiCount` 0..(total-1); `roomVisibility` ∈ {`open`, `closed`}; в team всегда 4 игрока.

### 2.1 1х01 (FFA, 2 игрока)
- Игрок пропал → бот доигрывает. Оппонент играет против бота.
- Если пропали **оба** → завершить по текущим очкам (решение #5), без бот-вс-бот.
- Это главный случай изменения поведения: раньше был мгновенный форфейт против одного оппонента.

### 2.2 3 игрока (FFA)
- 1 ушёл → бот держит место, матч продолжается для 2 человек + бот.
- 2 ушли → 1 человек против 2 ботов — продолжаем.
- Все 3 ушли → завершить по очкам.
- **Важно:** в мультиплеере проверь текущую логику форфейта (`findTimeoutWinner`/`endRoundByTimeoutForfeit`): как сейчас определяется победитель, когда один из 3+ игроков таймаутится? Захват перехватывает ЭТОТ путь ДО форфейта, поэтому матч просто продолжается штатно.

### 2.3 4 игрока (FFA, сам за себя) + выбор места
- Выбор места (`choose_seat`) — ТОЛЬКО до старта (`seatSelectionRequired = !gameActive && totalPlayers>2 && !areAllHumanPlayersSeated()`).
- Захват работает только при `gameActive === true`, поэтому с выбором места он НЕ пересекается. Дисконнект во время выбора места → старая лобби-логика, бота нет.
- После старта `seatIndex` фиксирован → бот играет на том же `seatIndex`, ничего не пересаживаем.
- 1–3 ушли → боты держат их места, матч идёт. Все 4 ушли → по очкам.

### 2.4 2ѐ2 (team)
- Бот играет за КОМАНДУ ушедшего (команда = `playerOrder index % 2`, см. `matchResultPayload`/`roomStatePayload`).
- Результат матча начисляется по команде: оба человека команды получают win/loss. Т.к. userId сохранён — ушедший игрок всё равно в participants и получает результат своей команды.
- **Нюанс качества:** `ai.js` (`AIPlayer.chooseMove`) НЕ знает про команды — играет «за себя» (минимизирует свою руку/максимизирует очки), не подыгрывает партнёру. Это то же поведение, что у обычных fill-ботов, но в 2ѐ2 это заметно слабее человека. Приёмлемо для v1, но зафиксировать как known limitation.
- Если у команды оба игрока ушли — команду ведут два бота; матч продолжается, пока есть хотя бы один человек за столом. Все 4 ушли → по очкам.

### 2.5 Открытые vs закрытые комнаты
- `joinable` возможен только при `!gameActive` (см. `livePresence.getRoomSnapshot`). После старта посторонний НЕ может занять место бота.
- Реклейм места — только исходный игрок (`resume` проверяет `requestUserId === player.userId`). Никто другой не заберёт место.
- Открытые/закрытые различаются только до старта (видимость в лобби). На захват не влияют.
- `isWaitingInOpenRoom` (ожидание в открытой комнате) — pre-game, захвата нет.

## 3. Модель триггеров (когда именно подключается бот)

Два пути отсутствия, оба должны вести к захвату:

1. **Таймаут хода (основной, 30с = TURN_TIMEOUT_MS).** Игрок на своём ходу не ходит 30с (бездействие или обрыв) → `handleTurnTimeout` → вместо форфейта `botTakeover.begin(player, reason)`.
2. **Истечение reconnect grace (120с = RECONNECT_GRACE_MS).** Игрок отключился и НЕ на его ходу; grace истёк → `finalizeReconnectTimeout` → вместо форфейта `begin(player, "disconnect")`.

`reason`: `"idle"` если сокет жив, иначе `"disconnect"`.

### Стык с pause/resume таймера — ОСТОРОЖНО
Сейчас есть `pauseTurnTimerForReconnect` / `resumeTurnTimerAfterReconnect`. Если при дисконнекте на своём ходу таймер ставится на паузу «навсегда», то 30с никогда не сработают и захвата не будет. Нужно:
- Либо НЕ ставить таймер хода на паузу при включённом флаге (пусть идёт и в 30с срабатывает захват),
- Либо ввести отдельный таймер ABSENCE_TAKEOVER_MS=30000 от момента отсутствия.
Проверь точное поведение этих функций перед правкой. Цель: «по окончании видимого обратного отсчёта бот подключается» (требование владельца).

## 4. Возврат человека и гонки

Два пути возврата:
1. **Реконнект** (токен `dominoRoomReconnectionToken`, `onJoin` reuse / `completeReconnectGrace`): если `isBotControlled(player)` → `resume()`.
2. **Кнопка «Вернуть управление»** (онлайн, но был простой таймаут бездействия): сообщение `resume_control`.

Гонки, которые НАДО закрыть:
- **Человек вернулся ровно в момент хода бота.** Защита через существующий `turnVersion` + `actionId`: один ход = один актор. Бот и человек не должны сходить дважды. resume не должен откатить уже сыгранный ход.
- **Бот уже сходил, потом человек вернулся** — ход бота валиден и остаётся; человек продолжает со следующего хода. Показать «бот сыграл за вас, пока вас не было».
- **Дребезг (flaky сеть).** `minBotControlMs` (3с) не даёт мгновенно реклеймить и тут же снова отвалиться.
- **Два одновременных resume_control** — идемпотентность: второй вызов ничего не делает (`isBotControlled` уже false).

После resume: `scheduleTurnTimer()` заново (человеку — полное окно), отменить запланированный ход бота.

## 5. «Ни одного человека за столом» (решение #5, обобщённо на все режимы)

Условие: после `begin()` нет НИ ОДНОГО подключённого человека (все человеческие места либо под ботом, либо изначальный AI). Тогда:
- НЕ гонять бот-вс-бот до конца.
- Завершить матч по текущим очкам: `gameOverReason = "all_absent"`.
- **Определить победителя явно** (нужен `winnerIndex`/`winnerKey` для `buildPlatformMatchPayload`): по наибольшим очкам (FFA) или по командным очкам (team).
- **Ничья:** если очки равны — определить tie-break (напр. по roundWins, или официальная ничья без списания ставки). Это продуктовое решение — уточнить у владельца, по умолчанию: ничья = возврат ставок.
- **Экономика:** это НОРМАЛЬНОЕ завершение (`matchOutcome="normal"`/`"all_absent"`), НЕ форфейт. Не вызывать `settleForfeitStakeForRoom` — идти по обычному расчёту.

## 6. Персистентность состояния — КРИТИЧНО (легко упустить!)

Сейчас `schemaStateSnapshot.js` НЕ сохраняет новые поля. Без правки после рестарта сервера / Redis-restore захваченное место «забудет» про бота и снова уйдёт в grace/форфейт.

### 6.1 `buildSchemaStateSnapshotData` — в маппинг игрока добавить:
```js
controller: player?.controller || "human",
takeoverActive: player?.takeoverActive || false,
takeoverReason: player?.takeoverReason || "",
takeoverSince: player?.takeoverSince || 0,
```

### 6.2 `buildRestoredPlayerRows` — добавить поля И исправить `isConnected`:
```js
controller: String(entry?.controller || "human"),
takeoverActive: Boolean(entry?.takeoverActive),
takeoverReason: String(entry?.takeoverReason || ""),
takeoverSince: Number(entry?.takeoverSince || 0),
// Было: isConnected: Boolean(entry?.isBot)
// Стало (захваченное место тоже «активно», играет бот):
isConnected: Boolean(entry?.isBot) || Boolean(entry?.takeoverActive),
```
Иначе после restore захваченное место будет `isConnected=false` и логика снова пойдёт в grace.

### 6.3 Проверить восстановление в `DominoRoom.js`
Там, где `buildRestoredSchemaStateData` применяется к state и создаются Player-схемы — перенести 4 поля в объекты Player. Иначе они останутся дефолтными.

## 7. Результат матча и экономика (решение #2 — реальные ставки)

- Захваченное место — рядовой участник (`participants`), результат win/loss начисляется как обычно.
- Игрок НЕ попадает в `forfeitUserIds`/`forfeitPlayerIds` (он не сдавался — за него играл бот).
- `matchOutcome` остаётся `"normal"` (кроме all_absent).
- Ставки: при старте уже зарезервированы (`reserveEconomyStakeForRoom`). При захвате→нормальном завершении идёт ОБЫЧНЫЙ расчёт, НЕ `settleForfeitStakeForRoom`. Проверь, что путь завершения после захвата не уходит в форфейт-расчёт.
- Антифрод: нельзя «отключиться, чтобы не проиграть» (бот доигрывает, реальный результат) и нельзя «отключиться, чтобы гарантированно выиграть» (бот «medium»).

---

## 8. Presence / лобби-счётчики

- `livePresence` отдельный от schema (Redis, TTL 180с, stale>90с). Когда человек отвалился, его presence пропадает → `connectedPlayers` комнаты падает. Это влияет только на счётчики/лобби, НЕ на геймплей.
- Комната с захваченными местами не становится joinable (т.к. gameActive). OK.
- Не трогать `setRoomGameActive` логику.

---

## 9. Клиент / UI по режимам

- Бейдж 🤖 над местом, где `player.controller === "bot"`. Маппинг места по `seatIndex`/`seatNumber` (важно для 3/4 игроков, где места расставлены по кругу).
- Кнопка «Вернуть управление» — только для СВОЕГО места (свой userId). В team нельзя реклеймить место партнёра.
- Тосты `bot_takeover`/`bot_resume` — текст уже приходит в `msg`. В team можно уточнить формулировку («ваш партнёр / соперник»).
- Реакции/гифты/голос отсутствующего не шлются (сокета нет). voiceEnabled для него должен сняться.
- Спектаторы/поздние подключения в идущую игру невозможны (joinable=false).

## 10. Анти-абьюз и честность

- `maxTakeoversPerMatch` (8) на userId: при превышении `begin()` вернёт false → откат к старому форфейту (защита от бесконечного «ушёл-вернулся» ради затягивания).
- `botDifficulty="medium"` (решение #3).
- Сброс счётчиков `resetForNewMatch()` на старте каждого матча.

## 11. Чек-лист «не сломать» (регрессии)

- [ ] При `DOMINO_BOT_TAKEOVER` выкл. — поведение 100% как сейчас (все ветки `begin()/resume()` возвращают false → fall through).
- [ ] Явный «Выйти» (`explicit_leave`) — ПО-ПРЕЖНЕМУ мгновенный форфейт (решение #1). Проверять `explicitLeaveSessionIds` ПЕРЕД захватом.
- [ ] reconnect grace (120с) и timeout-continue не сломаны для обычных сценариев без флага.
- [ ] Расчёт ставок (reserve/settle) не изменён для обычных и форфейт-матчей.
- [ ] `turnVersion`/`actionId` защита от двойного хода работает для бота.
- [ ] Схема Colyseus: 4 новых поля Player с дефолтами — обратная совместимость клиента (старый клиент игнорирует неизвестные поля).
- [ ] Снапшот/restore переносит новые поля (раздел 6).
- [ ] team-режим: порядок/команды/результат не изменились (userId/seatIndex/playerOrder не трогаем).
